# Local Release Scope

이 문서는 GitHub 업로드 승인이 아니라 로컬 배포 초안의 허용 범위와 차단 범위를
고정하는 명세입니다. 저장소 생성, remote 연결, push는 별도 사용자 승인 전까지
금지합니다.

## 허용 대상

- 범용 공고·출처·지원 상태 데이터 모델
- 사용자 설정형 직무·지역·플랫폼·점수 기준
- 합성 회사·공고·사용자 예시
- 공고별 맞춤 문안 수정, revision 스냅샷과 체크섬 충돌 방지
- 이력서·공고·사용자 허용 범위 교집합에 따른 동적 맞춤 항목
- 학력·경력·자격 사실 잠금과 공고에 명시된 지원서 질문만 추가
- 품질 재검사, 승인 무효화, 원자적 PDF 생성과 제출본 동결
- 질문-only 승인 차단, 섹션별 최소 작성 기준과 지원서 답변 별도 파일
- 로컬 전용 DB·산출물 경로 보호
- 기능·격리·보안 자동 검사와 GitHub Actions 정의

## 금지 대상

- 실제 이름, 전화번호, 이메일, 주소와 계정 식별 정보
- 실제 이력서, 포트폴리오, 경력 증빙, 지원 패키지와 지원 이력
- API 키, AI 토큰, 쿠키, 세션, 비밀번호와 Telegram 자격증명
- 개인용 SQLite DB, 로그, 보고서, 스크린샷과 절대 경로
- 특정 개인의 직무·회사·연봉·지역·플랫폼 우선순위
- 개인용 LaunchAgent, 포트 `8765`, 개인용 DB 참조

## 개인용 기능에서 범용화한 항목

| 개인용에서 검증된 안전 동작 | 로컬 배포 초안 |
|---|---|
| 직접 수정과 여러 산출물 동기화 | DB, JSON, Markdown, HTML 동시 저장 |
| 오래된 화면의 덮어쓰기 차단 | `expectedChecksum` 불일치 시 `409` |
| 수정 전 버전 보관 | `package_revisions`와 `.revisions/` 스냅샷 |
| 저장 실패 시 원상복구 | 파일 교체와 DB 트랜잭션 롤백 테스트 |
| 수정 후 품질 재판정 | 사용자 설정 점수 기준으로 상태 재계산 |
| 승인 뒤 수정 시 PDF·승인 무효화 | 이전 PDF 보관, 승인 이력 `invalidated` 기록 |
| 승인 PDF 원자적 생성 | 임시 PDF 생성, 형식·페이지 검증 후 교체 |
| 제출 준비 이후 무결성 유지 | 별도 제출 경로·체크섬·페이지 수 동결 |
| 마감·제외 공고 보관 | 활성 공고와 보관함 필터 분리 |
| SQLite 시간 표시 보정 | UTC 저장값을 `Asia/Seoul`로 명시 변환 |
| 낮은 적합도 경고 | `search.yml`의 `scoring.review_below` 사용 |
| 개인 직무 전용 고정 문항 | 직무 중립 섹션과 공고별 질문 기반 동적 편집 |

## 의도적으로 이관하지 않은 항목

- 개인 이름을 포함한 PDF 파일명과 개인 이력서 레이아웃
- 특정 직무군 전용 자기소개 문항·점수표·회사 가점
- 실제 지원 플랫폼 자동 로그인·자동 제출
- 개인 Codex·Claude 토큰 저장 또는 대시보드 내 AI 계정 연결
- Telegram 알림 코드·설정·자격증명
- 개인용 DB 마이그레이션과 기존 지원 데이터 변환

## 승인 게이트

`npm run verify`와 pre-publish 안전 검사가 통과해도 업로드 권한은 생기지 않습니다.
GitHub 저장소 생성·연결·push 직전에 사용자가 범위와 파일 목록을 다시 확인하고
명시적으로 승인해야 합니다.

## Private 개발 단계의 알려진 제한

- 이미 생성한 맞춤 패키지는 기본 이력서나 공고별 질문이 바뀌어도 자동으로 다시
  만들지 않습니다. private 개발 단계에서는 새 버전 갱신 흐름을 후속 작업으로
  관리하며, 공개 전에는 변경 감지와 사용자 확인 기반 갱신 기능을 추가합니다.
- CodeQL은 private 저장소에서 실행하지 않고 일반 테스트와 로컬 보안 게이트를
  사용합니다. 저장소를 공개하거나 GitHub Code Security를 사용할 때 활성화됩니다.
